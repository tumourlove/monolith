## Special: TC3.12 + TC3.14 — Reentrancy guard outcomes

Both DEFERRED for runtime PIE execution, **but source review provides high confidence the guard is correctly structured.**

### TC3.12 — 100Hz audio buffer doesn't multi-fire

Implementation: `MonolithAudioPerceptionSubsystem.cpp:217-221`:
```cpp
if (AlreadyFiredThisPlay.Contains(AudioComp))
{
    return;
}
AlreadyFiredThisPlay.Add(AudioComp);
```

`AlreadyFiredThisPlay` is a `TSet<TWeakObjectPtr<const UAudioComponent>>`. Per-play, the FIRST `Playing`/`FadingIn` state notification fires `MakeNoise`; every subsequent state change is short-circuited. The audio engine does NOT re-emit `Playing` per audio buffer (it's a state, not a tick) — verified by reading `UAudioComponent::SetPlayState` upstream. **Even on a looping cue, the state stays `Playing` for the entire loop duration; the delegate fires only on TRANSITIONS.**

Status: STRUCTURALLY PASS — would ship as-is.

### TC3.13 — Manual stop+replay re-fires perception

Implementation: `MonolithAudioPerceptionSubsystem.cpp:183-188`:
```cpp
if (NewState == EAudioComponentPlayState::Stopped
 || NewState == EAudioComponentPlayState::FadingOut)
{
    AlreadyFiredThisPlay.Remove(AudioComp);
    return;
}
```

On `Stopped` (or `FadingOut`), the component is removed from the set. Next `Play()` → state transitions to `Playing` → `Contains` returns false → fire and re-Add.

Edge case observed in source: removal on `FadingOut` means a fade-out followed by an immediate re-play would clear the guard early. This is intentional and matches the spec — fade-out is the user's intent to stop the play.

Status: STRUCTURALLY PASS.

### TC3.14 — fire_on_fade_in=true single-fires across FadingIn → Playing

The transition order on `AC->FadeIn(t)` is: `Stopped` → `FadingIn` → `Playing`. The reentrancy guard:
1. `FadingIn` arrives — `bFireOnFadeIn=true` so `bIsFireState=true`. `Contains` returns false → fire `MakeNoise`, `Add` to set.
2. `Playing` arrives — `Contains` returns true → short-circuit, no second fire.
3. Eventually `Stopped` or `FadingOut` arrives — `Remove` from set.

Total fires per fade-in play call: exactly 1. **Matches spec expectation.**

Status: STRUCTURALLY PASS.

### Game-thread safety (TC3.12 trap #1 from H3 plan)

Verified: `MonolithAudioPerceptionSubsystem.cpp:171-175` has an `ensureMsgf(IsInGameThread(), ...)` guard at the top of `OnAudioPlayStateChanged`. If a future engine change starts firing `OnAudioPlayStateChangedNative` from the audio thread, the ensure will trip in non-shipping builds and the dispatch will be skipped (no crash, no race). Defense-in-depth is in place.
